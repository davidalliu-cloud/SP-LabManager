# SARP LAB Management System

Professional first-version web app for a construction materials testing laboratory. The implemented scope focuses on concrete cube testing:

Sample received -> test created -> technician enters data -> test completed -> document controller prepares report -> chief approves -> PDF print/download -> report issued.

Sample codes are generated by received month, for example `2026-01-001` for the first January 2026 sample and `2026-02-001` for the first February 2026 sample. The Sample Register and New Sample form use client codes only; the Clients page is where the head of lab can add clients, assign a client code, and see which client/project/contact details belong to each code.

The concrete compressive strength test entry screen follows the current Albanian worksheet structure: register details, casting/received/testing dates, equipment, lab environmental conditions, and a batch table for multiple cube specimens in one test file.

Concrete compressive strength reports are split automatically because the report template supports a maximum of 3 cube specimens per report. For example, 10 tested cubes generate 4 draft reports: 3 + 3 + 3 + 1.

Large concrete cube deliveries can be scheduled into multiple test batches under one sample. For example, 60 received cubes can be registered once, then scheduled as 20 cubes at 7 days and 40 cubes at 28 days. Each schedule row creates its own test alert and due date, and the sample status becomes `Partially Tested` until all scheduled batches are completed.

## App Structure

- `app/` - Next.js App Router pages for dashboard, sample register, new sample, tests, reports, monthly summary, clients, projects, delayed items, users, and settings.
- `app/employees` - employee management for technicians and engineers involved in sample retrieval, testing, reporting, and approvals.
- `components/` - reusable layout, status badges, summary cards, workflow cards, tables, notifications, and report preview.
- `lib/` - typed lab model, calculations, status helpers, Supabase client stub, seed data, and workflow store.
- `supabase/schema.sql` - production-ready table structure, enums, indexes, numbering helpers, and starter RLS policies.

## Environment Variables

Create `.env.local`:

```bash
NEXT_PUBLIC_SUPABASE_URL=your-supabase-project-url
NEXT_PUBLIC_SUPABASE_ANON_KEY=your-supabase-anon-key
SUPABASE_SERVICE_ROLE_KEY=your-service-role-key-for-server-jobs-only
NEXT_PUBLIC_APP_URL=http://localhost:3000
```

The current first build uses local demo state so you can validate the workflow before connecting CRUD calls to Supabase.

## Supabase Setup

1. Create a Supabase project.
2. Open SQL Editor and run `supabase/schema.sql`.
3. Create a private storage bucket named `reports` for approved PDF files.
4. Enable email authentication in Supabase Auth.
5. Add users in Supabase Auth, then insert matching rows into `public.users`.

Example first admin row after creating the auth user:

```sql
insert into public.users (id, full_name, email, role)
values ('AUTH_USER_UUID', 'First Admin', 'admin@example.com', 'Admin / Managing Director');
```

## Run Locally

```bash
npm install
npm run dev
```

Open `http://localhost:3000`.

## Build

```bash
npm run build
npm start
```

## Deploy to Vercel

1. Push this project to GitHub.
2. Import the repository in Vercel.
3. Add the environment variables listed above.
4. Deploy.
5. In Supabase Authentication settings, add your Vercel domain to allowed redirect URLs.

## First Admin User

1. Create the admin user in Supabase Auth.
2. Copy the auth user UUID.
3. Insert into `public.users` with role `Admin / Managing Director`.
4. Sign in with that user once authentication screens are connected.
